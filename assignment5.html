Challenge 5: Accessibility Research (and good practices)

I successfully completed this challenge in assignment four. Here I will add additional findings and commentary.

Semantics are super important when coding in HTML. Gone are the days of endless div tags and chopped up images laid out in a table. It's these semantics that inform the browser and screen readers how to display and navigate the page. After all, HTML was designed to communicate the structure of a page, so it's important we use these semantics correctly.

Other basics:

  • When marking up an image, it's important to use alt tags and even labels - so a vision impaired user can understand what is being presented.
  • Do NOT use tables for arranging the page or placement of pictures. (This includes the archaic practice of slicing up images.) Tables exist for tabular data, and screen readers and browsers interpet them as such.
  • Never use images for words/text. This is because the browser or screen reader will not read the image as text.
  • Use nav tags! The <nav> element makes it clear to screen readers that these links are for navigating the current site.
  • <SPAN> and <DIV> tags give no information to the browser or screen reader, and should be used infrequently.
  • If there isn't a predefined tag to accomplish your task, add a "role" and name it.
  • Visually impaired users sometimes rely on clear contrast to be able to depict what they see.
  • Developers/designers don't have to do it on our own! - Tools exist to help quantify contrast ratios, such as Contrast Ratio.com.
  • Do NOT open a link in a new window (unless you explicitly inform the user first). You do not want to surprise the user or crowd their browser with open tabs.
  • In the example image of Challenge 4: Replicate a Google Search Page", the links are not underlined. According to W3schools.com, color should not be used as the ONLY visual indicator of meaning.

The primary takeaway here is that misusing HTML markup will result in misinterpretation by screen readers and confused users. Be a smart developer and be kind - which means being considerate and inclusive for all users.

Extra Challenge: Explore

Do you see the hand wave emoji on the first line of the above contact form? 👋 I found a full list of emojis which are included as part of charset UTF-8. This adds a whim of fun to the text and can be read by all browsers using UTF-8 even though it appears as a small image.

Another attribute I added for this assignment is the pattern attribute to validate the email address in the form. There was conflicting advice on StackOverflow regarding the best RegEx to use for this, and I didn't take the time to review the RegEx in detail. I chose this regex:
< input type="email" pattern="[a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,4}$" />
Even though this was an interesting find, the authors also noted that type="email" will basically do the validation for you even without using the pattern attribute.

Did you notice the change in font for the Regular Expression I printed above? I found the HTML code element, shown as the opening tag <code> and closing tag </code> will format this text without having to set a custom style.

Finally, I discovered how to escape HTML characters so the browser can print them without interpreting. I used the text representation of a left bracket instead of typing '<', since the browser has a specialized use for brackets.
